# Furigana Tooltip — Developer Guide

## What this is

A Chrome extension (Manifest V3) that shows furigana (phonetic hiragana readings) in a hover tooltip when the user selects Japanese kanji text. The core constraint: **zero page modification**. The tooltip is a CSS overlay injected at the `<html>` level, never into page content.

---

## Repository layout

```
src/
  utils.js            Pure functions: kana conversion, detection, HTML rendering
  content_script.js   Main content script: tokenizer, tooltip DOM, event wiring
tooltip.css           Tooltip styles with all:initial CSS isolation
scripts/
  copy-vendor.js      Copies kuromoji browser build + dict files → vendor/
  generate-icons.js   Generates icons/icon{16,48,128}.png in pure Node.js
  pack.js             Validates build artifacts and zips for Chrome Web Store
test/
  utils.test.js       Jest unit tests for all pure functions in src/utils.js

--- generated by npm install / npm run build ---
vendor/               kuromoji.js + 13 dict .dat.gz files (~20 MB)
icons/                icon16.png, icon48.png, icon128.png
content_script.js     esbuild IIFE bundle of src/content_script.js + src/utils.js
```

The files in the dashed section are **build artifacts** — gitignored, regenerated by `npm install`.

---

## Build pipeline

```bash
npm install          # installs deps + copies vendor files + generates icons + builds
npm run build        # esbuild src/content_script.js → content_script.js (IIFE)
npm test             # jest unit tests (node environment, no Chrome APIs needed)
npm run pack         # build + zip for Chrome Web Store → furiganize.zip
```

**Always edit files in `src/`**, never the root `content_script.js` (it gets overwritten by `npm run build`).

## Architecture

### Data flow

```
user selects text
  → mouseup fires on document
  → handleSelectionEnd() reads window.getSelection().toString()
  → isJapanese() gates on presence of Japanese characters
  → rangeRect() gets viewport coordinates of the selection
  → tooltip shows loading state at that position
  → loadTokenizer() resolves (lazy, singleton, ~1s on first call)
  → kuromoji.tokenize() returns token array with katakana readings
  → buildHTML() converts tokens → <ruby>/<span> HTML string
  → tooltip innerHTML updated, repositioned
```

### Key design decisions

**`all: initial` CSS isolation**
The tooltip `<div>` uses `all: initial` to hard-reset every CSS property to its spec-initial value before applying extension styles. This is the only reliable way to prevent arbitrary page stylesheets from bleeding into the tooltip. Without it, pages set global `font-size`, `line-height`, `color`, `font-family`, etc. that would break the furigana layout.

**Appended to `<html>`, not `<body>`**
`document.documentElement.appendChild(tooltipEl)` keeps the tooltip outside any `<body>` transforms/filters. CSS `position: fixed` is supposed to be viewport-relative but gets re-parented by ancestor `transform`, `filter`, or `will-change: transform` properties — a common pattern on animated landing pages. Attaching to `<html>` avoids body-level re-parenting.

**Symbol-based async cancellation**
`handleSelectionEnd()` mints a new `Symbol` → `activeToken` on every call. After the async tokenizer await, it checks `activeToken === token` before touching the DOM. This makes superseded requests silently abort rather than overwriting a newer tooltip.

**Single tokenizer promise**
`tokenizerPromise` is a module-level variable. The first selection starts dictionary parsing; every subsequent call — including concurrent ones during the ~1s load — awaits the same promise. No duplicate parsing.

**`rangeRect()` fallback**
`Range.getBoundingClientRect()` returns `{width:0, height:0}` for selections that span block-level element boundaries (two `<p>` tags, `<li>` to `<p>`, etc.). The fallback to `getClientRects()[last]` gets the rect of the final line of the selection, which is always a valid position for tooltip placement.

**No scroll/selectionchange hide triggers**
Earlier versions hid the tooltip on scroll and `selectionchange`. Both were too aggressive: trackpad momentum micro-scrolls dismissed the tooltip before the user finished reading; some pages' own JavaScript temporarily collapsed the selection (e.g. custom copy handlers), causing flicker. The tooltip now only hides on `mousedown` outside it or `Escape`.

## Testing

Tests in `test/utils.test.js` cover the five pure functions from `src/utils.js`. They run in Node (no DOM, no Chrome API mocking needed).

```bash
npm test
npm test -- --coverage    # coverage report
npm test -- --watch       # re-run on file change
```

What's covered:
- `toHiragana` — standard katakana, small kana (ッ→っ), boundary cases
- `hasKanji` / `isJapanese` — positive/negative for each character class
- `esc` — XSS payloads in surface_form and reading fields
- `buildHTML` — ruby for kanji, span for particles, span for unknown words, skipping redundant ruby on pure kana tokens, multi-token output

What's not covered (intentional):
- Tokenizer loading (requires kuromoji dict files — integration territory)
- Tooltip DOM construction (requires browser DOM)
- Event wiring (requires browser events)

## Known limitations

### IPAdic dictionary coverage
kuromoji ships with IPAdic (~2007). Compound words not in the dictionary are split into individual morphemes, losing phonological changes in compound form. Example: `特化` → `特(とく) + 化(か)` instead of `とっか`. Common vocabulary (発表, 学校, etc.) is well covered; technical and academic compounds may show per-character readings.

### CSS transform re-parenting
Pages that apply `transform` to `<html>` itself (rare but possible) will still misposition the tooltip. No practical workaround short of using a separate extension page rendered via `chrome.windows` or `chrome.tabs`.

## Chrome Web Store submission

1. `npm test` — all tests must pass
2. `npm run pack` — creates `furiganize.zip`
3. Upload at `https://chrome.google.com/webstore/devconsole`
4. Required: a privacy policy URL. Since the extension collects nothing, a single-page GitHub Pages or Notion doc stating "Furigana Tooltip processes selected text locally in the browser. No data is transmitted externally." satisfies the requirement.

## Expansion paths

### Better dictionary coverage
Replace IPAdic with UniDic for broader compound coverage. UniDic requires custom compilation but is supported by kuromoji via the `dicPath` option (same interface). UniDic dict files are ~50 MB vs the current ~20 MB.

### Settings / options page
Add `chrome.storage.sync`-backed options (`options.html` + `options.js`). Manifest addition:
```json
"options_ui": { "page": "options.html", "open_in_tab": false },
"permissions": ["storage"]
```
Useful settings: per-site disable list, font size, color theme, toggle furigana-for-katakana.

### Per-site disable
Read a blocked-origins list from `chrome.storage.sync` at content script startup. If `window.location.origin` is in the list, skip all `addEventListener` calls. Requires the `"storage"` permission.

### Keyboard shortcut toggle
Add `"commands"` to manifest.json, listen in a service worker, and send a `chrome.tabs.sendMessage` to the content script toggling a module-level `enabled` flag. No new permissions needed.

### Network fallback for unknown compounds
When kuromoji splits a run of consecutive kanji-only `KNOWN` tokens, optionally POST the combined surface to jisho.org's API to fetch the compound reading. Requires explicit user opt-in (selected text sent externally).
